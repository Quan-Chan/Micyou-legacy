## 1. 手机断连后桌面端无限加载，无法重新连接

**状态：** ✅ 已提交（`5608fdd`）

### 文件

- `AudioEngine.jvm.kt`
- `NetworkServer.kt`

### 根因

两个互不相关的 bug 共同导致：

1. **JitterBuffer 序列号未重置：** 断连后 `expectedSequenceNumber` 停留在 ~5000，新连接的 UDP 包 seqNum~0 被当成乱序包丢弃，音频管道无数据输出，UI 一直显示加载。
2. **`stopAudioProcessing()` 未在 Connecting 状态触发：** 协程条件判断遗漏了 `StreamState.Connecting`，音频处理协程残留，`startAudioProcessing()` 因判断已运行而跳过，管道无法重启。

### 修复

| 文件 | 改动 |
|---|---|
| `NetworkServer.kt:256-260` | `handleConnection()` 开头调用 `jitterBuffer.stop()` + `jitterBuffer.start()` 重置序列号 |
| `AudioEngine.jvm.kt:140` | `stopAudioProcessing()` 的条件增加 `StreamState.Connecting` |

## 3. EQ 零值显示异常（+0 和 0 显示不同）

**状态：** ✅ 已提交（`5608fdd`）

### 文件

`DesktopSettings.kt`

### 根因

显示逻辑 `if (gain >= 0) "+${gain.toInt()}"` 让 `0f` 显示为 `+0`，而 `-0.4f`（`toInt()` 截断为 0）走 else 显示为 `0`。两个文本不同，且因实际浮点值不同滑块位置也不同。

### 修复

```kotlin
val displayVal = gain.toInt()
if (displayVal > 0) "+$displayVal" else "$displayVal"
```

所有 `toInt()=0` 的值都统一显示为 `"0"`，不再区分 +0 和 0。

## 4. EQ 拖完滑块后滚轮操作让修改失效

**状态：** ✅ 已提交（`5608fdd`）

### 文件

`DesktopSettings.kt`

### 根因

`pointerInput(Unit)` 的 key 是 `Unit`，协程只启动一次，捕获了**首次组合时的** `gain` 和 `eqConfig` 引用（闭包快照）。用户拖动滑块后状态更新、Compose 重组，但滚轮协程未重启，仍用旧值调用 `setEqualizerConfig`→**把拖动结果覆盖回了旧值**。

### 修复

用 `rememberUpdatedState` 包裹 `gain` 和 `eqConfig`，协程内每次读取最新值。分两处：

| 位置 | 改动 |
|---|---|
| 前置放大器（~1775-1791） | 添加 `currentPreAmpEqConfig`，滚轮内用 `currentPreAmpEqConfig.preAmp` |
| 频段滑块（~1823-1841） | 添加 `currentGain` + `currentEqConfig`，滚轮内用二者 |

## 5. 首次点击"开始串流"按钮动画卡死

**状态：** ✅ 已提交（`5608fdd`）

### 文件

`AudioEngine.jvm.kt`

### 根因

`startLocked()` 中 `mdnsAdvertiser.reAdvertise()` 在 Swing EDT（UI 线程）**同步执行**。内部涉及 DNS 反向解析（`InetAddress.getLocalHost()`）和 mDNS socket 创建，阻塞 300~700ms。期间 Compose 无法渲染帧，动画完全冻结。

### 修复

```kotlin
// 之前（EDT 同步调用）
runCatching { mdnsAdvertiser.reAdvertise(port, bindAddress) }

// 之后（后台协程异步执行）
scope.launch {
    runCatching { mdnsAdvertiser.reAdvertise(port, bindAddress) }
}
```

网络服务器已在 EDT resume 前启动完毕，mDNS 只用于服务发现，不阻塞连接。

## 7. EQ 滑块连续性不一致（松手吸附整数）

**状态：** 🔄 未提交（最新改动）

### 文件

`DesktopSettings.kt`

### 根因

滑块连续（无 steps），但显示和判定用 `toInt()`（整数）。值 0.3f 和 -0.3f 都显示 0，但滑块位置不同，导致"相同数值、不同高度"的诡异感。

### 修复

添加 `onValueChangeFinished`，松手时通过 `round()` 吸附到最近整数。拖动过程保持连续不变。

| 位置 | 改动 |
|---|---|
| 前置放大器 | 添加 `onValueChangeFinished = { eqConfig.copy(preAmp = round(...)) }` |
| 每个频段 | 添加 `onValueChangeFinished = { round(eqConfig.gains[index]) }`，仅值变化时更新 |

## 8. EQ 调到 0 后文字仍加粗

**状态：** ✅ 已提交（`5608fdd`）

### 文件

`DesktopSettings.kt`

### 根因

样式判断用 `gain != 0f`，但浮点精度问题导致滑块拖回零时实际值是 0.000001f，`!= 0f` 为 true，文字保持加粗/主色。

### 修复

```kotlin
// 之前
if (gain != 0f) FontWeight.Bold else FontWeight.Normal

// 之后
if (displayVal != 0) FontWeight.Bold else FontWeight.Normal
```

颜色和粗细统一用整数取整值判断。

## 9. 快速点击"开始串流"按钮，后台排队一直触发

**状态：** 🔄 未提交（最新改动）

### 文件

`AudioStreamViewModel.kt`

### 根因

每次 `startStream()` 丢一个协程到 `viewModelScope`，这些协程在 `startStopMutex.withLock` 排队等候。用户快速点 10 次 Start，虽被 `isStartStreamRequestPending` 挡住，但最终总有 1 个协程在后台执行完，且 Stop→Start→Stop→Start 会让标记一直 stuck。

### 修复

用 `streamJob`（`Job?`）替代 `isStartStreamRequestPending`：

| 行 | 改动 |
|---|---|
| 第 106 行 | `private var streamJob: Job? = null` |
| `startStream()` | 每次先 `streamJob?.cancel()` 取消旧的，再 `viewModelScope.launch { }` 创建新的 |
| `startStream()` | 捕获 `CancellationException` 时恢复 Idle |
| `stopStream()` | 加 `streamJob?.cancel()` 清除等待中的启动 |

**效果：** 每次新 Start/Stop 直接中断上一个排队中的 Start，不积累、不排队。

## 10. 串流时进入设置页严重卡顿

**状态：** 🔄 未提交（最新改动）

### 文件

- `MainViewModel.kt`
- `DesktopHomeEnhanced.kt`

### 根因

`AppUiState` 包含 `audioMetrics`（每音频包更新，50-100Hz）和 `metricsHistory`（每 500ms），每次更新触发 `_uiState.update()` → 设置页整个 recompose 50-100 次/秒。唯一需要使用它们的组件是 `MonitoringPanel`。

### 修复

| 文件 | 行 | 改动 |
|---|---|---|
| `MainViewModel.kt` | ~146-147 | 从 `AppUiState` 移除 `audioMetrics` 和 `metricsHistory` |
| `MainViewModel.kt` | ~277-289 | 简化 `combine`：不再合并 metrics/history，只合并 `audioStreamViewModel.uiState` |
| `MainViewModel.kt` | ~372-373 | 删除 `copy` 中的 `audioMetrics =` / `metricsHistory =` |
| `MainViewModel.kt` | 第 188 行后 | 新增 `val metricsHistoryFlow` |
| `DesktopHomeEnhanced.kt` | ~263-266 | `MonitoringPanel` 改为通过 `viewModel.audioMetricsFlow.collectAsState()` 独立订阅 |

**效果：** `AppUiState` 不再按音频包速率更新，设置页只在用户操作时 recompose。

## 11. EQ 预设卡片滚轮翻预设

**状态：** ✅ 已修复

### 文件

`DesktopSettings.kt`

### 问题描述

鼠标光标在 EQ 预设卡片内滚动滚轮时，页面上下滚动，预设列表不动。超出可见区域的预设只能拖拽，无法通过滚轮浏览。

### 修复方案

使用 `onPointerEvent(PointerEventType.Scroll)` 在预设卡片外层 `Box` 上拦截滚轮事件，通过 `rememberLazyListState` + `scrollToItem` 实现前后翻预设。

### 改动细节

| 改动 | 说明 |
|---|---|
| `val presetRowState = rememberLazyListState()` | 创建 `LazyRow` 的滚动状态 |
| `val coroutineScope = rememberCoroutineScope()` | 协程作用域用于 `scrollToItem` |
| `LazyRow(state = presetRowState)` | 绑定滚动状态到 `LazyRow` |
| `.onPointerEvent(PointerEventType.Scroll) { ... }` | 外层 Box 拦截滚轮 |
| 内部：`scrollDelta.y < 0 → scrollToItem(index-1)` | 往上滚 → 往左翻 |
| 内部：`scrollDelta.y > 0 → scrollToItem(index+1)` | 往下滚 → 往右翻 |
| `event.changes.forEach { it.consume() }` | 阻止事件冒泡到父层 `LazyColumn` |
